### VAL-CR1 — per-video caption check (A11Y-07)

- **Correctness:** PASS
  The regex `<video[^>]*>(?:(?!</video>).)*?<track[^>]*\bkind=["']captions["'][^>]*/?>`
  with `re.DOTALL | re.IGNORECASE` correctly handles: self-closing `<track />`,
  `<track>` without slash, tracks on different lines (including when `>` lands on a
  new line due to multi-line attribute lists), tracks preceded by other elements
  (source, comments), and the `default` boolean attribute. The tempered greedy token
  `(?:(?!</video>).)*?` is the canonical safe pattern and does not backtrack
  catastrophically on realistic lesson HTML. The 0-video auto-pass path (line 476-477)
  is correct and unchanged. The original false negative — two tracks both nested in
  video 1 while video 2 has none — now correctly counts 1 of 2 and emits FAIL.

- **Test coverage:** PASS with one caveat (LOW severity — see below)
  `TestValidatorFalseNegativeFixes.test_a11y07_catches_unbalanced_tracks` exercises
  the exact false-negative scenario: `minimal-fail.html` has two `<video>` elements,
  both caption tracks inside the first, none in the second. The old total-count path
  would have passed (2 tracks, 2 videos); the new per-video path correctly fails.
  The test is non-tautological.

- **Edge cases:** No crash on malformed HTML (unclosed `<video` tag returns 0 matches,
  triggering FAIL or auto-pass depending on whether the element was parsed). No crash
  when `:root` block is absent. The only theoretical gap is a `<track>` whose closing
  `>` is separated from `kind="captions"` by a newline AND whose `kind` attribute
  appears after a `>` on the same line as the opening `<track` — verified working.

### VAL-CR2 — selector boundary tightening (CLR-05/CLR-06)

- **Correctness:** PASS
  The final regex (after `32af782`):
  `\.gold(?![\w-])[^{]*\{[^}]*(?:^|[;{\s])color\s*:\s*var\(\s*--gold\s*\)`
  with `re.IGNORECASE | re.MULTILINE` correctly:
  - Matches `.gold { color: var(--gold); }` — FAIL as intended
  - Matches `.gold, .highlight { color: var(--gold); }` — FAIL as intended
  - Does NOT match `.gold-border { border-left-color: var(--gold); }` — PASS as intended
  - Does NOT match `.area-card.gold-card .area-icon { color: var(--gold); }` — PASS as intended
  - Does NOT match `.gold { background-color: var(--gold); }` — PASS as intended
  - Does NOT match `.gold { border-color: var(--gold); }` — PASS as intended
  All 9 boundary cases verified in-process. CLR-06 uses the identical structural
  pattern for `.accent`/`--accent` and passes the same boundary analysis.

- **Test coverage:** PASS
  `TestCLR05Boundaries` (4 unit tests added in `32af782`) directly exercises the
  two false-positive cases from the overreach and the two cases that must still fail.
  These tests import the validator module directly and construct minimal HTML, making
  them precise and isolated — the strongest test form for regex boundary verification.
  `TestValidatorFalseNegativeFixes.test_clr05_catches_grouped_selector` is redundant
  with the existing `TestValidatorFailingFixture.test_clr05_catches_gold_text` since
  both run on `minimal-fail.html` and assert the same CLR-05 FAIL output. This is
  LOW severity — redundancy is harmless — but the `TestCLR05Boundaries` class is
  the authoritative coverage for this fix.

- **Edge cases:** One confirmed false negative exists for minified CSS with no
  whitespace after the opening brace: `.gold{color:var(--gold)}` does NOT match.
  This is a pre-existing structural limitation of the `(?:^|[;{\s])` property
  boundary anchor — the `{` was already consumed by the literal `\{` in the regex
  pattern, leaving no boundary character before `color`. In practice, all lesson CSS
  is authored with formatters that produce at least one space or newline after `{`,
  so this gap is LOW risk and does not warrant blocking the merge. It is worth noting
  so the regex can be hardened (e.g., replace `[;{\s]` with `[;\s]` and add `{?\s*`
  after the `\{` literal) in a follow-up.

### VAL-CR3 — inline style scan (CLR-02)

- **Correctness:** PASS
  The inline scan regex `style=["'][^"']*#([0-9a-fA-F]{3,8})\b[^"']*["']` correctly
  catches `style="color: #dc2626"` (double-quoted) and `style='color: #dc2626'`
  (single-quoted). It does NOT match `data-color="#dc2626"` or hex values in text
  content (e.g., inside `<code>` blocks) because the pattern requires the `style=`
  attribute prefix. Canonical hex values in `style=` attributes (e.g.,
  `style="color: #007baf"`) are correctly filtered by the `CANONICAL_HEX_SET` check
  downstream. Multi-property style values (e.g., `style="font-family: Arial; color:
  #dc2626"`) are correctly caught. The regex produces the captured hex without the
  leading `#`, which is then re-prefixed before the non-canonical check — correct.

- **Test coverage:** PASS
  `TestValidatorFalseNegativeFixes.test_clr02_catches_inline_style_hex` asserts
  CLR-02 FAILs on `minimal-fail.html`, which contains
  `<div style="color: #dc2626">` at line 145. This is a direct test of the new
  inline scan path. `#dc2626` is both non-canonical AND in `PROHIBITED_COLORS`, so
  CLR-08 would also flag it; the test confirms CLR-02 specifically fires. Non-tautological.

- **Edge cases:** No false positive risk for hex values inside `<code>` elements,
  comments, or `data-*` attributes — all confirmed. The one theoretical false positive
  is a `style=` attribute containing both single and double quotes (e.g., an escaped
  inner quote), but this is not valid HTML in lesson files and is not a realistic
  concern.
